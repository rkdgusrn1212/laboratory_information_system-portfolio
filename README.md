### 1.2. 프로젝트 주제

## 3. 나의 프로젝트 기여 활동 내역

### 3.1. 일정 및 자원 관리

- 전체 일정을 주 단위로 계획, 각 주마다 주어지는 Todo List 를 각 팀원의 역량에 맞춰 분담시키고 각 업무 산출물들을 Issue에 업로드 하게함. Issue에 남긴 기록을 토대로 업무 성과를 확인하고 성취율을 추후 계획에 반영.
- git project의 milestone과 todo list 기능을 활용하여, 각 업무의 진행 종료 상황을 한눈에 파악하고 공유할 수 있게 함.

### 3.2. 공통 개발환경 구성

- 나의 개발 경험과 현업의 공통분모가 존재하는 기술들 위주로 구성하면서, 기술의 버전을 헬스케어솔루션사업본부에서 사용하는 특정 버전으로 맞춤.

### 3.3. 기획/설계

우선, 내 담당 파트와는 별개로 기획/설계를 총괄하여 파트 병합 및 최종 검토, 문서화를 수행하였다.

#### 3.3.1. 화면설계

인증, 무인접수, 진료 파트를 설계하고 팀원들의 타 파트 설계 결과물들을 최종 검수하고 하나로 병합하는 과정을 수행했다. 

#### 3.3.2. DB 설계

- 1차적으로 내가 먼저 전체 ERD를 대략적으로 작성하고 기본적인 네이밍 컨벤션과 데이터형 지정 규칙을 작성.
- 2차적으로 나와 각 팀원이 자신이 맞은 파트에 대한 스키마 검토 및 수정을 수행, 나는 여기서 인증, 무인접수, 진료 파트 그리고 진료의 기초가되는 처방 마스터, 진료행위코드 마스터, 검체 코드, 검체 용기 코드 마스터등의 기본 마스터성 테이블 까지 내가 담당하여 검토 및 수정을 수행.
- 3차적으로 내가 2차 작업내용을 병합하고, 물리적인 성능을 고려하여 반정규화를 수행. 또한 필요에 따라 트리거와 뷰를 추가 생성함.

##### 3.3.2.1. 파트별 설계 내역

**인사 파트**

- 인사 파트 내부 프로세스 정의 : 프로젝트 시작 당시 요구사항으로 주어진 정보를 바탕으로 사용자와 그 사용자의 권한과 필요한 프로세스를 상세 정의함.

  > 주어진 요구사항
  >
  > 1. 의사는 프로그램의 모든 프로세스를 수행 가능하다.
  > 2. 간호사는 검사 처방 이후의 프로세스만 수행 가능하다. (채혈 접수 부터 가능하다)
  > 
  > <br>당시 정의한 프로세스 목록
  > 
  > 1. 직원 등록 프로세스.
  > 2. 직원 탈퇴 프로세스.
  > 3. 직원 정보 변경 프로세스.
  > 4. 직원 인증 프로세스.(로그인/로그아웃)
  > 5. 점유 인증 프로세스.(점유 인증된 채널을 통해 직원 등록, 비밀번호 변경 등을 수행)
  > 6. 비밀번호 찾기 프로세스.

- 인사 파트 화면 설계 : 인사 파트에서 가능한 모든 프로세스들의 기능을 사용할 수 있는 UI/UX를 구상.
- 인사 파트 DB 설계 : 정의한 프로세스를 구현할 수 있는 논리/물리 스키마를 작성. 의사와 간호사를 슈퍼-서브 타입으로 반정규화 되었고 간호사가 별도의 컬럼을 가지지 않아 스키마에서 제거된 모습이 특징이다.

**무인 접수 파트**

- 무인 접수 파트 내부 프로세스 정의 : 사실 환자는 기본 요구사항에서는 그냥 DB에 직접 올린 데이터를 사용해도 된다고 했는데, 본인 파트 개발이 타 팀원에 비해 조기 종료되서 추가적으로 진행한 파트이다. 그래서 최대한 단위 시스템 내에 Actor가 적은 셀프접수를 목표로 했다. 
Actor가 오직 환자 하나다.

 > 당시 정의한 프로세스 목록
 >
 > - 환자 등록 : 환자를 진료 접수시키기 위해 먼저 환자를 병원 DB에 등록시키는 과정으로 환자, 방문&예약 접수에 앞서서 환자 주민번호를 이용해 환자의 등록여부를 조회 미등록 환자를 별도의 등록 프로세스로 보내 등록을 완료한 후 기존 접수 프로세스를 진행하게 한다.
 > - 방문 접수 : 당일 진료를 보기위한 접수로 접수시 현재 의사/진료과별 대기현황을 알려줄 수 있어야하며 방문 접수됨과 동시에 진료 파트의 진료 대기 환자 조회에서 접수 시간과 함께 순서대로 조회가 가능해야 한다.
 > - 예약 접수 : 원하는 날짜에 진료를 보기위한 접수로 원하는 날짜와 시간을 선택할 수 있어야하며, 접수 시 진료 파트의 진료 대기 환자 목록에서 당일 날짜의 예약 접수를 확인 가능해야한다.

## 4. 아쉬운점

### 4.1 프로젝트 완성도

#### 4.1.1. Redux Toolkit + RTK Query

Redux Toolkit + RTK Query 사용 환경을 구성해서 팀원 들에게 배포했으나 팀원들이 활용을 아예 못한 점이 아쉽다. 팀원들이 공부를 병행했지만 활용까지는 실패하고 자신들의 컴포넌트 내에서 axios로 처리했다고 한다. 결과적으로 팀장인 나 혼자 Redux Toolkit+RTK Query를 통해 상태관리와 데이터 패칭을 구현했기 때문에 내가 개발한 기능 내에서만 활용되는 등의 활용성이 제한되었다. 그래도 내가 개발한 기능에 대해서는 개발 효율성이 올라갔으니 프로젝트 단위로 봐도 없는 것보다는 효과적이었다고 할 수 있다. 

#### 4.1.2. Typescript

Typescript도 Redux Toolkit처럼 사용 환경을 구성해서 팀원들에게 배포했으나 팀원들이 학습하고 적용하는데까지는 역량이 부족하여 실패하고 결국 나혼자 사용함. 그래도 나 혼자임에도 불구하고 타입스크립트가 전체 프론트엔드 소스에서 꽤나 큰 비율을 차지하는데 이는 프론트엔드에서 내가 작성한 코드가 절반 이상을 차지하기 때문이다. 그러므로 프로젝트의 프론트엔드 개발에 있어서 꽤나 큰 효과를 봤지만, 팀원들 파트까지 타입스크립트로 구성 못한 것은 아쉽다.

### 4.2. 사서 고생

#### 4.2.1. 갑작스러운 하드웨어 고장
발표 전날에 로컬 서버의 하드웨어 고장으로 하드웨어 수리하겠다고 컴퓨터 해체하고 고장 원인을 분석하고 수리하려 했다. 서버 컴퓨터가 계속 부팅이 안되고 어쩌다 부팅 성공했을때 OS영역만 로드되면 블랙 스크린이 되었다. 어쩌다 부팅 성공했을 때, 안전모드를 통한 OS 재설치도 시도해 봤으나 소용이 없어서, 하드웨어 문제로 판정하고 하드디스크 교체를 시도하려 했으나 그를 위한 얇은 드라이버도 없었기에 밤이 되어서야 오늘 해결 불가한 문제라고 결론 지었다. 따라서, 클라우드 서버 배포로 목표를 변경, AWS EC2 올리기위해 밤샘 작업했다.
